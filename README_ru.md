## Подробно о системе тактирования
> **Почему сначала изучать тактирование?** Потому что почти все периферийные устройства зависят от тактовой частоты для работы. Настройка тактирования — самый базовый и важный шаг в встроенной разработке. Неправильная конфигурация приведёт к ненормальной работе периферии, неточному бодрейту USART, невозможности перечисления USB и другим проблемам.
>

### Обзор дерева тактирования STM32F1
Система тактирования STM32F103 очень гибкая, имеет множество источников тактовой частоты и делителей. Вот упрощённое дерево тактирования:

```plain
                          ┌─────────────┐
                          │   HSE       │  Внешний высокочастотный кварц (плата DKX: 8 МГц)
                          │  8 MHz      │
                          └──────┬──────┘
                                 │
                          ┌──────▼──────┐
                          │   HSI       │  Внутренний RC-генератор
                          │  8 MHz      │  (низкая точность, ±1%, быстрый запуск)
                          └──────┬──────┘
                                 │
                 ┌───────────────┼───────────────┐
                 │               │               │
                 │         ┌─────▼─────┐         │
                 │         │   PLL     │  ФАПЧ    │
                 │         │  Умножитель│  ×2~×16 │
                 │         └─────┬─────┘         │
                 │               │               │
          ┌──────▼──────┐ ┌──────▼──────┐        │
          │ SYSCLK      │ │  USBCLK     │        │
          │ Системная    │ │  Тактовая   │        │
          │ макс 72MHz   │ │  частота USB│        │
          └──────┬──────┘ │  нужен 48MHz│        │
                 │        └─────────────┘        │
        ┌────────┼────────┐                      │
        │        │        │                      │
   ┌────▼───┐ ┌──▼───┐ ┌──▼────┐          ┌─────▼─────┐
   │ AHB    │ │ APB1 │ │ APB2  │          │  SYSCLK   │
   │ шина   │ │ шина  │ │ шина  │          │  выбор    │
   │≤72MHz  │ │≤36MHz│ │≤72MHz │          │  источника│
   └───┬────┘ └──┬───┘ └──┬────┘          │ HSI/HSE/PLL
       │         │        │               └───────────┘
  ┌────▼───┐ ┌───▼────┐ ┌─▼──────┐
  │ Cortex │ │USART2/3│ │USART1  │
  │ SysTick│ │TIM2-4  │ │SPI1    │
  │ DMA    │ │I2C1/2  │ │ADC1/2  │
  │ Flash  │ │SPI2    │ │TIM1    │
  └────────┘ │USB     │ │GPIO    │
             └────────┘ └────────┘
```

**Основные понятия: PLL (фазовая автоподстройка частоты — ФАПЧ)**

```plain
Тактовая частота PLL = Входная тактовая частота PLL × Коэффициент умножения PLL

Если выбран HSE в качестве входа PLL:
  PLLCLK = HSE × коэффициент умножения (×2 ~ ×16)

Пример (кварц 8 МГц на плате DKX):
  HSE × 9  = 8 × 9  = 72 МГц ← максимальная системная частота
  HSE × 6  = 8 × 6  = 48 МГц ← необходима для USB
  HSE × 4  = 8 × 4  = 32 МГц

Если выбран HSI в качестве входа PLL:
  PLLCLK = HSI × 2 × коэффициент умножения / 2
  PLLCLK = HSI × коэффициент умножения (×2 ~ ×16)
  Но HSI должен быть сначала разделён на 2 перед входом в PLL
```

### Подробно об источниках тактирования
#### HSI (High Speed Internal) — внутренний высокочастотный генератор
```plain
Характеристики:
├── Частота: 8 МГц (RC-генератор, есть температурный дрейф)
├── Точность: ±1% (заводская калибровка), дрейфует при изменении температуры
├── Преимущества: не требует внешних компонентов, работает сразу при включении
├── Недостатки: низкая точность, не подходит для USB, CAN, точного бодрейта
└── По умолчанию: при включении автоматически является источником системной частоты
```

**Когда использовать HSI?**

+ Простые сценарии вроде мигания светодиода, обработки кнопок, где не нужна точная частота
+ Резервный вариант при выходе из строя внешнего кварца
+ Сценарии быстрого запуска (HSI запускается быстрее HSE)

#### HSE (High Speed External) — внешний высокочастотный генератор
```plain
Характеристики:
├── Частота: 4-16 МГц (плата DKX использует кварц 8 МГц)
├── Точность: ±0.005% (зависит от качества кварца)
├── Преимущества: высокая точность, подходит для USB, CAN, точного бодрейта USART
├── Недостатки: требуется внешний кварц, время запуска (несколько сотен микросекунд~несколько миллисекунд)
└── Плата DKX: пассивный кварц 8 МГц + 2 конденсатора нагрузки 20 пФ
```

**Когда использовать HSE?**

+ Нужна функция USB (**обязательно** использовать HSE или HSE через PLL)
+ Нужна шина CAN (нужна точная частота)
+ Нужен точный бодрейт USART
+ Нужна максимальная частота 72 МГц

#### LSE (Low Speed External) — внешний низкочастотный генератор
```plain
Характеристики:
├── Частота: 32.768 кГц (для RTC)
├── Точность: высокая (малый температурный дрейф кварца)
├── Назначение: часы реального времени (RTC), сторожевой таймер
└── Плата DKX: возможно, кварц LSE не распаян (нужно проверить схему)
```

#### LSI (Low Speed Internal) — внутренний низкочастотный генератор
```plain
Характеристики:
├── Частота: около 40 кГц (неточно)
├── Назначение: независимый сторожевой таймер (IWDG), резервный источник RTC
└── Точность: низкая (±30%)
```

#### PLL (Phase Locked Loop) — фазовая автоподстройка частоты (ФАПЧ)
PLL — ядро системы тактирования, используется для умножения низкой частоты до высокой.

```plain
┌─────────────────────────────────────────────────┐
│                    PLL подробно                  │
├─────────────────────────────────────────────────┤
│                                                  │
│  Выбор входного источника:                       │
│  ┌──────┐    ┌─────────┐                         │
│  │ HSI/2│───►│         │    ┌───────────┐        │
│  └──────┘    │ PLL MUX │───►│  ÷ PLLMUL │──► PLLCLK
│  ┌──────┐───►│         │    │  (×2~×16) │        │
│  │ HSE  │    └─────────┘    └───────────┘        │
│  └──────┘                                        │
│                                                  │
│  Популярные конфигурации:                        │
│  ┌──────────┬──────────┬──────────────┐          │
│  │ Входная  │Коэфф.    │Выходная      │          │
│  │ частота  │умножения │частота       │          │
│  ├──────────┼──────────┼──────────────┤          │
│  │ HSI 8MHz │ ×9       │ 36 MHz*     │          │
│  │ HSE 8MHz │ ×9       │ 72 MHz ✓    │          │
│  │ HSE 8MHz │ ×6       │ 48 MHz ✓    │          │
│  │ HSE 8MHz │ ×4       │ 32 MHz ✓    │          │
│  └──────────┴──────────┴──────────────┘          │
│                                                  │
│  * HSI сначала делится на 2 (=4 МГц), затем ×9 = 36 МГц │
│                                                  │
└─────────────────────────────────────────────────┘
```

### Подробно о тактировании шин
```plain
                         SYSCLK (системная тактовая частота)
                              │
                ┌─────────────┼─────────────┐
                │             │             │
           ┌────▼────┐  ┌────▼────┐  ┌─────▼─────┐
           │  AHB    │  │  APB1   │  │   APB2    │

           │ ÷HPRE  │  │ ÷PPRE1 │  │  ÷PPRE2  │

           └────┬────┘  └────┬────┘  └─────┬─────┘
                │            │              │
                │            │              │
          ┌─────▼─────┐ ┌────▼────┐  ┌──────▼──────┐
          │Core, DMA  │ │USART2/3 │  │  USART1     │
          │SysTick    │ │I2C1/2   │  │  SPI1       │
          │Flash      │ │SPI2     │  │  ADC1/2     │
          │GPIO A~D   │ │TIM2-4   │  │  TIM1       │
          │           │ │USB      │  │  EXTI       │
          │           │ │CAN      │  │  AFIO       │
          └───────────┘ └─────────┘  └─────────────┘
```

#### Шина AHB — высокоскоростная шина
| Параметр | Описание |
| --- | --- |
| Максимальная частота | 72 МГц |
| Предделитель | SYSCLK ÷ 1/2/4/8/16/64/128/256/512 |
| Подключённые устройства | Ядро Cortex-M3, DMA, Flash, GPIO, SysTick |
| Метод настройки | `rcc::Config::hsi().hclk(72.MHz())` |


**Примечание:** Тактовая частота SysTick поступает от AHB (если SysTick настроен на использование процессорной тактовой частоты) или AHB/8.

#### Шина APB1 — шина низкоскоростных периферийных устройств
| Параметр | Описание |
| --- | --- |
| Максимальная частота | **36 МГц** (жёсткое ограничение, превышение повредит микросхему) |
| Предделитель | AHB ÷ 1/2/4/8/16 |
| Подключённые устройства | USART2, USART3, I2C1/2, SPI2, TIM2-4, USB, CAN |
| Метод настройки | `rcc::Config::hsi().pclk1(36.MHz())` |


**Важно:** Если предделитель APB1 > 1, то тактовая частота таймера = APB1 × 2.

#### Шина APB2 — шина высокоскоростных периферийных устройств
| Параметр | Описание |
| --- | --- |
| Максимальная частота | **72 МГц** |
| Предделитель | AHB ÷ 1/2/4/8/16 |
| Подключённые устройства | USART1, SPI1, ADC1/2, TIM1, GPIOA~D, EXTI, AFIO |
| Метод настройки | `rcc::Config::hsi().sysclk(72.MHz()).pclk2(72.MHz())` |


#### Тактовая частота АЦП
| Параметр | Описание |
| --- | --- |
| Максимальная частота | **14 МГц** |
| Источник тактовой частоты | APB2 ÷ 2/4/6/8 |
| Метод настройки | `rcc::Config::hsi().adcclk(14.MHz())` |


#### Тактовая частота USB
| Параметр | Описание |
| --- | --- |
| Требуемая частота | **48 МГц** (должна быть точной) |
| Источник тактовой частоты | Выход PLL (PLLCLK ÷ 1 или 1.5) |
| Требования к настройке | SYSCLK должен быть 48 МГц или 72 МГц |

### Настройка тактирования в stm32f1xx-hal
Библиотека HAL использует паттерн **Builder** для настройки тактирования, очень наглядно:

#### Основное использование
```rust
use stm32f1xx_hal::{pac, prelude::*, rcc};

fn main() {
    let dp = pac::Peripherals::take().unwrap();
    let mut flash = dp.FLASH.constrain();  // Настройка циклов ожидания Flash

    // Способ 1: компактный метод настройки
    let mut rcc = dp.RCC.freeze(
        rcc::Config::hsi()           // Использовать внутренний RC 8 МГц
            .sysclk(64.MHz())        // Системная частота 64 МГц
            .pclk1(32.MHz())         // Частота APB1 32 МГц
            .pclk2(64.MHz())         // Частота APB2 64 МГц
            .adcclk(8.MHz()),        // Частота АЦП 8 МГц
        &mut flash.acr,
    );

    // Способ 2: использование внешнего кварца + PLL
    let mut rcc = dp.RCC.freeze(
        rcc::Config::hse(8.MHz())    // Использовать внешний кварц 8 МГц
            .sysclk(72.MHz())        // PLL умножение до 72 МГц
            .pclk1(36.MHz())         // Деление APB1 до 36 МГц
            .pclk2(72.MHz())         // APB2 без деления
            .adcclk(14.MHz()),       // АЦП 14 МГц
        &mut flash.acr,
    );
}
```

#### Методы builder для `rcc::Config`
```rust
// Все доступные методы настройки (* — часто используемые)
rcc::Config::hsi()                    // Выбрать HSI как источник тактирования
rcc::Config::hse(8.MHz())            // Выбрать HSE как источник, указать частоту

.sysclk(72.MHz())    *               // Установить целевую системную частоту
.pclk1(36.MHz())    *                // Установить целевую частоту APB1
.pclk2(72.MHz())    *                // Установить целевую частоту APB2
.adcclk(14.MHz())   *                // Установить частоту АЦП
.hclk(72.MHz())                       // Установить частоту AHB (обычно = SYSCLK)

// PLL автоматически рассчитает коэффициент умножения на основе целевой частоты
// Не нужно устанавливать вручную!
```

#### Получение информации о тактовой частоте после freeze
```rust
let rcc = dp.RCC.freeze(
    rcc::Config::hse(8.MHz()).sysclk(72.MHz()),
    &mut flash.acr,
);

// Получение фактических частот
rprintln!("SYSCLK: {}", rcc.clocks.sysclk());   // Системная частота
rprintln!("HCLK:   {}", rcc.clocks.hclk());     // Частота AHB
rprintln!("PCLK1:  {}", rcc.clocks.pclk1());    // Частота APB1
rprintln!("PCLK2:  {}", rcc.clocks.pclk2());    // Частота APB2
rprintln!("ADCCLK: {}", rcc.clocks.adcclk());   // Частота АЦП
rprintln!("USBCLK valid: {}", rcc.clocks.usbclk_valid()); // Действительна ли частота USB
```

#### Зачем нужен `flash.acr`?
Скорость чтения Flash ограничена. При системной частоте выше 24 МГц необходимо вставлять циклы ожидания:

| Системная частота | Циклы ожидания Flash |
| --- | --- |
| 0-24 МГц | 0 циклов ожидания |
| 24-48 МГц | 1 цикл ожидания |
| 48-72 МГц | 2 цикла ожидания |


`freeze()` автоматически устанавливает правильное количество циклов ожидания в зависимости от системной частоты.

#### Разница между `constrain()` и `freeze()`
```rust
// constrain() — ограничивает RCC, возвращает настраиваемый объект
// Используется для ручной настройки тактирования каждого периферийного устройства
let mut rcc = dp.RCC.constrain();

// freeze() — выполняет настройку тактирования и замораживает за один шаг
// Автоматически рассчитывает все параметры деления/умножения
let rcc = dp.RCC.freeze(
    rcc::Config::hse(8.MHz()).sysclk(72.MHz()),
    &mut flash.acr,
);
```

Обычно рекомендуется использовать `freeze()` — проще и труднее допустить ошибку.

#### Продвинутое использование: ручное указание коэффициентов деления/умножения
```rust
// Если вам нужен полный контроль над настройкой тактирования, используйте RawConfig
let rcc = dp.RCC.freeze(
    rcc::RawConfig {
        hse: Some(8_000_000),       // Частота HSE
        pllmul: Some(7),            // Коэффициент умножения PLL (×9, индексация с 0)
        hpre: rcc::HPre::Div1,     // Предделитель AHB = без деления
        ppre1: rcc::PPre::Div2,    // Предделитель APB1 = AHB ÷ 2
        ppre2: rcc::PPre::Div1,    // Предделитель APB2 = без деления
        usbpre: rcc::UsbPre::Div1_5, // Предделитель USB
        adcpre: rcc::AdcPre::Div2,  // Предделитель АЦП = APB2 ÷ 2
        ..Default::default()
    },
    &mut flash.acr,
);
```

### Популярные конфигурации тактирования
#### Конфигурация 1: Минимальная (HSI по умолчанию)
```rust
// По умолчанию при включении: HSI 8 МГц, PLL не используется
// SYSCLK = 8 МГц, APB1 = 8 МГц, APB2 = 8 МГц

// Или можно использовать freeze

```

**Подходит для:** мигание светодиода, обработка кнопок, тестирование GPIO и других простых сценариев
**Не подходит для:** USB, CAN, высокоскоростной USART

#### Конфигурация 2: HSI с умножением до 64 МГц
```rust
let mut rcc = dp.RCC.freeze(
    rcc::Config::hsi()
        .sysclk(64.MHz())         // HSI × 8 = 64 МГц
        .pclk1(32.MHz())          // APB1 = AHB ÷ 2

// Примечание: HSI можно умножить максимум до 64 МГц, не до 72 МГц
```

### 2.6 Частые ошибки настройки тактирования
#### Ошибка 1: APB1 превышает 36 МГц
```rust
// ❌ Ошибка! Максимум APB1 — 36 МГц
let mut rcc = dp.RCC.freeze(
    rcc::Config::hse(8.MHz())
        .sysclk(72.MHz())
        .pclk1(72.MHz()),  // Ошибка! Превышает 36 МГц
    &mut flash.acr,
);

// ✓ Правильно
let mut rcc = dp.RCC.freeze(
    rcc::Config::hse(8.MHz())
        .sysclk(72.MHz())
        .pclk1(36.MHz()),  // Правильно
    &mut flash.acr,
);
```

#### Ошибка 2: Неточная тактовая частота USB
```rust
// ❌ HSI не подходит для USB
let mut rcc = dp.RCC.freeze(
    rcc::Config::hsi().sysclk(48.MHz()),
    &mut flash.acr,
);
// Точность HSI ±1%, USB требует ±0.25%, приведёт к сбою перечисления

// ✓ Необходимо использовать HSE
let mut rcc = dp.RCC.freeze(
    rcc::Config::hse(8.MHz()).sysclk(48.MHz()),
    &mut flash.acr,
);
```

#### Ошибка 3: Забыт `flash.acr`
```rust
// ❌ Отсутствует параметр flash.acr
let rcc = dp.RCC.freeze(rcc::Config::hse(8.MHz()).sysclk(72.MHz()));
// Ошибка компиляции! freeze требует два параметра

// ✓ Правильно
let mut flash = dp.FLASH.constrain();
let rcc = dp.RCC.freeze(
    rcc::Config::hse(8.MHz()).sysclk(72.MHz()),
    &mut flash.acr,  // Обязательно передать!
);
```

#### Ошибка 4: SYSCLK превышает 72 МГц
```rust
// ❌ STM32F103 максимум 72 МГц
let mut rcc = dp.RCC.freeze(
    rcc::Config::hse(8.MHz())
        .sysclk(80.MHz()),  // Ошибка!
    &mut flash.acr,
);
// freeze автоматически выберет ближайшую частоту, не превышающую 72 МГц
```

#### Ошибка 5: Тактовая частота АЦП превышает 14 МГц

```rust
// Если pclk2 = 72 МГц и нет деления для АЦП, частота АЦП = 72/6 = 12 МГц ✓
// Если pclk2 = 72 МГц и АЦП без деления, частота АЦП = 72 МГц ✗
```

## Зажигаем первый светодиод
```rust
//! Мигание светодиода на выводе PC13 STM32F103C8T6
//!
//! Этот пример предполагает, что светодиод подключён к выводу PC13,
//! как на плате Blue Pill.
//!
//! Примечание: без дополнительного оборудования PC13 не следует использовать
//! для управления светодиодом напрямую. Подробности см. в разделе 5.1.2
//! справочного руководства. Однако на плате Blue Pill это не является проблемой.

// Запрет использования unsafe-кода для обеспечения безопасности
#![deny(unsafe_code)]
// Сообщаем компилятору Rust не использовать стандартную библиотеку (необходимо для встроенных систем)
#![no_std]
// Сообщаем компилятору Rust об отсутствии традиционной функции main, используется точка входа от cortex-m-rt
#![no_main]

// Импорт функции обработки паники: при неустранимой ошибке останавливает работу CPU
use panic_halt as _;

// Импорт утилит для неблокирующих операций, используется для обработки асинхронных операций
use nb::block;

// Импорт макроса точки входа, предоставляемого runtime cortex-m
use cortex_m_rt::entry;
// Импорт основных модулей библиотеки HAL
// pac: уровень доступа к периферии (Peripheral Access Crate), предоставляет доступ к регистрам
// prelude: предварительный импорт часто используемых trait для упрощения кода
// timer: модуль таймера
use stm32f1xx_hal::{pac, prelude::*, timer::Timer};

use rtt_target::{rprintln,rtt_init_print};

// Определение точки входа программы, заменяет стандартную функцию main
#[entry]
fn main() -> ! {

    rtt_init_print!();

    // Получение периферийных устройств ядра Cortex-M (SysTick, NVIC и т.д.)
    // Метод take() гарантирует, что периферия получена только один раз, предотвращая повторное использование
    let cp = cortex_m::Peripherals::take().unwrap();
    // Получение периферийных устройств STM32F103 (GPIO, таймеры, USART и т.д.)
    let dp = pac::Peripherals::take().unwrap();

    // Получение и настройка контроллера сброса и тактирования (RCC)
    // Метод constrain() преобразует исходную структуру RCC в высокоуровневую абстракцию HAL
    let mut rcc = dp.RCC.constrain();

    // Получение порта GPIOC и разделение на отдельные выводы
    // Метод split() гарантирует уникальность владения выводом, предотвращая одновременное управление одним выводом из нескольких функций
    let mut gpioc = dp.GPIOC.split(&mut rcc);

    // Настройка вывода PC13 в режим выхода с толкающим каскадом
    // Регистр crh используется для настройки старших 8 выводов порта (PC8-PC15)
    // Для младших 8 выводов (PC0-PC7) следует передавать регистр crl
    let mut led = gpioc.pc13.into_push_pull_output(&mut gpioc.crh);
    
    // Настройка системного таймера (SysTick) как счётчика, срабатывает с указанной частотой
    // counter_hz() настраивает SysTick в режим счётчика на основе частоты
    let mut timer = Timer::syst(cp.SYST, &rcc.clocks).counter_hz();
    
    // Запуск таймера, установка частоты срабатывания 1 Гц (раз в секунду)
    timer.start(4.Hz()).unwrap();

    // Основной цикл: ожидание срабатывания таймера и переключение состояния светодиода
    loop {
        // Блокирующее ожидание первого срабатывания таймера (через 1 секунду)
        block!(timer.wait()).unwrap();
        // Установка PC13 в высокий уровень, светодиод гаснет (на Blue Pill светодиод включается низким уровнем)
        led.set_high();
        rprintln!("OPEN THE LED");
        
        // Блокирующее ожидание второго срабатывания таймера (ещё через 1 секунду)
        block!(timer.wait()).unwrap();
        // Установка PC13 в низкий уровень, светодиод загорается
        led.set_low();
        rprintln!("LOW THE LED");
    }
}
```

Вторая версия — автоматическое чтение микросхемы

```rust
#![allow(clippy::empty_loop)]
#![deny(unsafe_code)]
#![no_main]
#![no_std]

use panic_halt as _;

use cortex_m_rt::entry;
use stm32f1xx_hal::{pac, prelude::*}; // pac = Peripheral Access Crate

#[entry]
fn main() -> ! {
    // Получение владения периферией (можно вызвать только один раз, повторный вызов возвращает None)
    let p = pac::Peripherals::take().unwrap();

    // Ограничение RCC (контроллер сброса и тактирования)
    // constrain() возвращает объект со всеми настраиваемыми тактовыми частотами
    let mut rcc = p.RCC.constrain();

    // Разделение порта GPIOC на отдельные объекты выводов
    // split() возвращает независимый дескриптор каждого вывода
    let mut gpioc = p.GPIOC.split(&mut rcc);

    // Выбор вывода и уровня в зависимости от модели микросхемы
    cfg_select! {
        feature = "stm32f100" => {
            // STM32F100: PC9 высокий уровень включает
            gpioc.pc9.into_push_pull_output(&mut gpioc.crh).set_high();
        }
        feature = "stm32f101" => {
            // STM32F101: PC9 высокий уровень включает
            gpioc.pc9.into_push_pull_output(&mut gpioc.crh).set_high();
        }
        _ => {
            // STM32F103 (включая вашу плату DKX): PC13 низкий уровень включает
            // PC13 на платах Blue Pill/DKX — общая анода, низкий уровень = горит
            gpioc.pc13.into_push_pull_output(&mut gpioc.crh).set_high(); // На самом деле на платах LC устанавливается высокий уровень для включения
        }
    }

    loop {} // Сохранение состояния светодиода неизменным
}
```

### Режим прерывания таймера RTIC
<font style="color:#DF2A3F;"><<Рекомендуется для разработки>> RTIC</font>

#### Сравнение RTIC и прерываний «голого железа»
##### I. Сравнение архитектуры
###### Прерывания «голого железа» (Bare Metal)
```rust
// Необходимо вручную определять обработчики прерываний, привязывать к таблице векторов
#[interrupt]
fn TIM1_UP() {
    // Ручной вход в критическую секцию (отключение прерываний)
    cortex_m::interrupt::free(|_| {
        // Весь код в одной функции, управление ресурсами — на программисте
        // Нет управления приоритетами, всё вручную включается/выключается
    });
}
```

###### Прерывания RTIC
```rust
// Декларативная привязка прерывания через #[task(binds = ...)]
#[task(binds = TIM1_UP, priority = 1, local = [led, timer])]
fn tick(cx: tick::Context) {
    // Ресурсы управляются фреймворком RTIC, безопасность гарантируется на этапе компиляции
    // Приоритеты автоматически управляются планировщиком RTIC
}
```

---

##### II. Основные различия
| Характеристика | Прерывания «голого железа» | RTIC |
| --- | --- | --- |
| **Управление ресурсами** | Ручная защита `critical section` | Автоматическое распределение на этапе компиляции, нулевые накладные расходы |
| **Управление приоритетами** | Ручные операции с регистрами NVIC | Декларативная настройка `priority = N` |
| **Общий доступ к данным** | Требуется `static mut` + `unsafe` | `#[shared]` + `Mutex`, безопасно на этапе компиляции |
| **Критическая секция** | Ручное включение/отключение прерываний | RTIC автоматически генерирует оптимальную критическую секцию |
| **Привязка прерываний** | Изменение `interrupt.rs` или `device.x` | `#[task(binds = TIM1_UP)]` — одна строка |
| **Переключение контекста** | Ручное сохранение/восстановление регистров | Аппаратное автоматическое сохранение в стек (Cortex-M) |
| **Организация кода** | Вся логика в одной функции прерывания | Каждая задача независима, ресурсы разделены |
| **Защита от взаимоблокировок** | Программист должен быть осторожен | Определение на этапе компиляции (протокол потолка приоритетов) |


---

##### III. Сравнение управления ресурсами
###### Способ «голого железа»: `static mut` + `unsafe`
```rust
// Глобальная изменяемая статическая переменная, требуется unsafe для доступа
static mut LED_STATE: bool = false;
static mut COUNT: u8 = 0;

#[interrupt]
fn TIM1_UP() {
    unsafe {
        if LED_STATE {
            // Управление светодиодом...
            LED_STATE = false;
        } else {
            // Управление светодиодом...
            LED_STATE = true;
        }
        COUNT += 1;
    }
}
```

**Проблемы:**

+ Ошибки в блоке `unsafe` не отслеживаются компилятором
+ Одновременный доступ нескольких прерываний к одной переменной приводит к гонке данных
+ Прерывания с высоким приоритетом могут прервать низкоприоритетное и нарушить целостность данных

###### Способ RTIC: `#[local]` привязка на этапе компиляции
```rust
#[task(binds = TIM1_UP, priority = 1, local = [led, led_state: bool = false, count: u8 = 0])]
fn tick(cx: tick::Context) {
    // Каждый ресурс привязан к этой задаче на этапе компиляции
    // Другие задачи не могут получить доступ, естественная защита от гонки данных
    if *cx.local.led_state {
        cx.local.led.set_high();
        *cx.local.led_state = false;
    }
    *cx.local.count += 1;
}
```

**Преимущества:**

+ Нулевой `unsafe`, компилятор гарантирует корректность
+ Привязка ресурсов определяется на этапе компиляции, нулевые накладные расходы
+ Доступ к ресурсам через `cx.local.xxx`

---

##### IV. Сравнение приоритетов и планирования
###### Способ «голого железа»: ручные операции с NVIC
```rust
// Необходимо вручную настраивать приоритеты
fn setup_timer_interrupt() {
    unsafe {
        // Установка приоритета прерывания TIM1 = 1
        // Нужно знать адрес регистров NVIC и битовые поля
        let nvic = &*cortex_m::peripheral::NVIC::ptr();
        // Сложные операции с регистрами...
    }
}
```

###### Способ RTIC: декларативные приоритеты
```rust
// priority = 1 — и готово
#[task(binds = TIM1_UP, priority = 1)]
fn tick(cx: tick::Context) { ... }

// Задача с высоким приоритетом может вытеснить задачу с низким
#[task(binds = USART1, priority = 2)]
fn serial(cx: serial::Context) { ... }
```

**Правила приоритетов RTIC:**

+ Чем больше число, тем выше приоритет
+ Задача с высоким приоритетом может вытеснить задачу с низким приоритетом
+ Задачи с одинаковым приоритетом не вытесняют друг друга

---

##### V. Сравнение общих ресурсов
###### Способ «голого железа»: ручная критическая секция
```rust
static mut SHARED_DATA: u32 = 0;

#[interrupt]
fn LOW_PRIORITY() {
    cortex_m::interrupt::free(|_| {
        unsafe { SHARED_DATA += 1; }
    });
}

#[interrupt]
fn HIGH_PRIORITY() {
    // Забыли добавить критическую секцию? Гонка данных! Компилятор не сообщит!
    unsafe { SHARED_DATA += 1; }
}
```

###### Способ RTIC: `#[shared]` + автоматическая блокировка
```rust
#[shared]
struct Shared {
    data: u32,
}

#[task(binds = TIM1_UP, priority = 1, shared = [data])]
fn tick(cx: tick::Context) {
    // lock() автоматически управляет критической секцией, безопасность на этапе компиляции
    cx.shared.data.lock(|d| {
        *d += 1;
    });
}

#[task(binds = USART1, priority = 2, shared = [data])]
fn serial(cx: serial::Context) {
    // Задача с высоким приоритетом обращается к тому же ресурсу,
    // RTIC автоматически генерирует оптимальную блокировку
    cx.shared.data.lock(|d| {
        *d += 1;
    });
}
```

**Механизм блокировок RTIC (протокол потолка приоритетов PCP):**

+ Когда задача с низким приоритетом удерживает блокировку, задача с высоким приоритетом ожидает
+ Но RTIC автоматически повышает приоритет задачи, удерживающей блокировку, чтобы задачи со средним приоритетом не «перепрыгивали»
+ Всё это определяется на этапе компиляции, нулевые накладные расходы

---

VI. Итоги

| Сценарий | Рекомендуемый способ |
| --- | --- |
| Изучение принципов прерываний | Голое железо (понимание низкоуровневых механизмов) |
| Разработка реального проекта | RTIC (безопасность, эффективность, удобство сопровождения) |
| Одно простое прерывание | Голое железо (примерно одинаковый объём кода) |
| Несколько прерываний + общие ресурсы | RTIC (защита от гонки данных) |
| Строгие требования к реальному времени | RTIC (абстракции с нулевыми накладными расходами, детерминированное планирование) |


**Основные преимущества RTIC:**

+ **Абстракции с нулевыми накладными расходами**: всё определяется на этапе компиляции, без дополнительных расходов при выполнении
+ **Безопасность на этапе компиляции**: гонки данных и взаимоблокировки обнаруживаются на этапе компиляции
+ **Декларативное программирование**: атрибутные макросы описывают «что делать», фреймворк генерирует «как делать»
+ **Протокол потолка приоритетов**: оптимальное управление критическими секциями

## Шина CAN (без проверки устройства — не тестировалось)
```markdown
use bxcan::Fifo;
use bxcan::filter::Mask32;

#[entry]
fn main() -> ! {
    let dp = pac::Peripherals::take().unwrap();
    let mut flash = dp.FLASH.constrain();

    // CAN требует внешний кварц для обеспечения точности тактовой частоты
    let mut rcc = dp.RCC.freeze(rcc::Config::hse(8.MHz()), &mut flash.acr);

    let mut can1 = {
        let gpioa = dp.GPIOA.split(&mut rcc);
        let rx = gpioa.pa11;  // CAN RX
        let tx = gpioa.pa12;  // CAN TX

        let can = dp.CAN.can(dp.USB, (tx, rx), &mut rcc);

        // Настройка битовой синхронизации: 125 кбит/с, точка выборки 87.5%
        bxcan::Can::builder(can)
            .set_bit_timing(0x001c_0003)
            .leave_disabled()
    };

    // Настройка фильтра (приём всех кадров)
    let mut filters = can1.modify_filters();
    filters.enable_bank(0, Fifo::Fifo0, Mask32::accept_all());
    drop(filters);

    // Включение CAN
    let mut can = can1;
    block!(can.enable_non_blocking()).unwrap();

    // Тест обратной петли: приём кадра и немедленная отправка обратно
    loop {
        if let Ok(frame) = block!(can.receive()) {
            block!(can.transmit(&frame)).unwrap();
        }
    }
}
```

**Выводы CAN (плата DKX):** PA11(CAN RX), PA12(CAN TX) — обратите внимание, общие выводы с USB



## USB последовательный порт (без проверки устройства — не тестировалось)
### USB последовательный порт с опросом (без проверки устройства — не тестировалось)
```rust
#![no_std]
#![no_main]

extern crate panic_semihosting;

use cortex_m::asm::delay;
use cortex_m_rt::entry;
use stm32f1xx_hal::usb::{Peripheral, UsbBus};
use stm32f1xx_hal::{pac, prelude::*, rcc};
use usb_device::prelude::*;
use usbd_serial::{SerialPort, USB_CLASS_CDC};

#[entry]
fn main() -> ! {
    let dp = pac::Peripherals::take().unwrap();
    let mut flash = dp.FLASH.constrain();

    // USB должен использовать системную частоту 48 МГц
    let mut rcc = dp.RCC.freeze(
        rcc::Config::hse(8.MHz()).sysclk(48.MHz()).pclk1(24.MHz()),
        &mut flash.acr,
    );

    assert!(rcc.clocks.usbclk_valid());  // Проверка действительности тактовой частоты USB

    let mut gpioc = dp.GPIOC.split(&mut rcc);
    let mut led = gpioc.pc13.into_push_pull_output(&mut gpioc.crh);
    led.set_high();  // Выключить светодиод

    let mut gpioa = dp.GPIOA.split(&mut rcc);

    // На линии USB D+ есть подтягивающий резистор
    // При разработке нужно притянуть D+ к земле для инициирования USB RESET
    let mut usb_dp = gpioa.pa12.into_push_pull_output(&mut gpioa.crh);
    usb_dp.set_low();                           // Притянуть D+ к земле
    delay(rcc.clocks.sysclk().raw() / 100);     // Кратковременная задержка

    // Настройка периферии USB
    let usb = Peripheral {
        usb: dp.USB,
        pin_dm: gpioa.pa11,                                  // USB DM = PA11
        pin_dp: usb_dp.into_floating_input(&mut gpioa.crh),  // USB DP = PA12
    };
    let usb_bus = UsbBus::new(usb);

    // Создание последовательного устройства CDC-ACM
    let mut serial = SerialPort::new(&usb_bus);

    // Сборка USB-устройства
    let mut usb_dev = UsbDeviceBuilder::new(&usb_bus, UsbVidPid(0x16c0, 0x27dd))
        .device_class(USB_CLASS_CDC)
        .strings(&[StringDescriptors::default()
            .manufacturer("Fake Company")
            .product("Serial port")
            .serial_number("TEST")])
        .unwrap()
        .build();

    loop {
        if !usb_dev.poll(&mut [&mut serial]) {
            continue;
        }

        let mut buf = [0u8; 64];
        match serial.read(&mut buf) {
            Ok(count) if count > 0 => {
                led.set_low();  // Включить светодиод
                // Преобразование полученных символов в верхний регистр и отправка обратно
                for c in buf[0..count].iter_mut() {
                    if 0x61 <= *c && *c <= 0x7a {
                        *c &= !0x20;  // 'a'~'z' → 'A'~'Z'
                    }
                }
                // Запись обратно (может потребоваться несколько записей)
                let mut write_offset = 0;
                while write_offset < count {
                    match serial.write(&buf[write_offset..count]) {
                        Ok(len) if len > 0 => {
                            write_offset += len;
                        }
                        _ => {}
                    }
                }
            }
            _ => {}
        }
        led.set_high();  // Выключить светодиод
    }
}
```

**Ключевые моменты USB:**

+ **Обязательна системная частота 48 МГц** (протокол USB требует точную тактовую частоту)
+ PA11 = USB D-, PA12 = USB D+
+ При разработке необходимо вручную инициировать USB RESET
+ VID/PID `0x16c0:0x27dd` — неофициальный тестовый идентификатор
+ Необходима сборка в режиме release (в режиме debug Flash переполнится)

### USB последовательный порт с прерываниями (без проверки устройства — не тестировалось)
Использование прерываний для обработки связи USB.

```rust
// Глобальный объект USB
static mut USB_BUS: Option<UsbBusAllocator<UsbBusType>> = None;
static mut USB_SERIAL: Option<SerialPort<UsbBusType>> = None;
static mut USB_DEVICE: Option<UsbDevice<UsbBusType>> = None;

#[entry]
fn main() -> ! {
    // ... Код инициализации USB ...

    // Включение прерываний USB
    unsafe {
        NVIC::unmask(Interrupt::USB_HP_CAN_TX);   // Высокий приоритет
        NVIC::unmask(Interrupt::USB_LP_CAN_RX0);  // Низкий приоритет
    }

    loop { wfi(); }  // Вся работа выполняется в прерываниях
}

#[interrupt]
fn USB_HP_CAN_TX() {
    usb_interrupt();
}

#[interrupt]
fn USB_LP_CAN_RX0() {
    usb_interrupt();
}

fn usb_interrupt() {
    let usb_dev = unsafe { USB_DEVICE.as_mut().unwrap() };
    let serial = unsafe { USB_SERIAL.as_mut().unwrap() };

    if !usb_dev.poll(&mut [serial]) {
        return;
    }

    let mut buf = [0u8; 8];
    match serial.read(&mut buf) {
        Ok(count) if count > 0 => {
            // Обработка полученных данных
            for c in buf[0..count].iter_mut() {
                if 0x61 <= *c && *c <= 0x7a {
                    *c &= !0x20;  // Преобразование в верхний регистр
                }
            }
            serial.write(&buf[0..count]).ok();
        }
        _ => {}
    }
}
```

**Опрос vs Прерывания:**

+ Опрос: просто, но CPU постоянно занят ожиданием
+ Прерывания: CPU может перейти в режим сна, экономия энергии

## ST7789 3D-куб (без цвета — каркас)
Отрисовка вращающегося 3D-куба на дисплее ST7789 240x240 с поддержкой цветных граней, каркасного рендеринга и счётчика FPS.

| Вывод дисплея | Вывод MCU | Описание функции |

| SDA | PA7 | Выход данных SPI (SPI1_MOSI) |
| DC | PA0 | Выбор команда/данные |
| RES | PA1 | Аппаратный сброс |
| CS | GND | Выбор кристалла — низкий уровень (всегда активен) |

+ Отображение счётчика FPS в реальном времени
+ Настраиваемый размер куба, угол обзора, скорость вращения
+ Тактовая частота SPI: 36 МГц
